📖 背景介绍

为什么需要 GenAI 可观测语义规范？

🤖

GenAI 新概念的涌现

随着 AI 快速发展，AI Agent 系统中涌现出 Model、Prompt、Token、Tool Calling、Agent、Memory、Session 等新的核心概念。这些概念需要像传统系统中的 HTTP 请求、数据库调用一样，被标准化地采集、展示和消费。

📊

可观测数据标准缺失

相关领域存在可观测数据采集标准缺失、口径不统一等问题。不同团队各自记录"模型名"、"输入长度"、"Token 数"等信息，字段命名和统计口径无法对齐。

🌍

OpenTelemetry 的推动

OTel 在 2024 年初开始推动 GenAI 语义规范建设，希望为这些新对象建立统一的数据采集规范——Semantic Conventions（SemConv），解决跨模型、跨框架、跨平台的观测标准问题。

"SemConv 是 OTel 的灵魂，推动其逐步完善并走向 Stable 是社区最重要的工作。"
— OTel 社区核心 Maintainer

统一数据语言

OTel GenAI SemConv 为核心概念提供标准化字段：

  • gen_ai.system
  • gen_ai.request.model
  • gen_ai.usage.input_tokens

不同业务、不同基础设施、不同观测后端共享同一套分析方法，真正做到"同一类问题用同一套数据解释"。

📈

支撑统一治理

  • 技术排查：通过 Trace ID 查看跨 Agent 完整链路，分钟级定位问题
  • 经营分析：效果数据跨业务可比，直接用于产品决策
  • 评测：真实用户轨迹持续积累，自动构建评测数据集
  • 合规：统一的审计链路，满足安全备案刚性要求

  • 无侵入埋点可复用
  • SDK 封装可复用
  • 平台分析可复用
  • 看板和告警策略可复用

✨ LoongSuite GenAI SemConv 增强内容

相较于 OTel GenAI SemConv 的核心增强点

🎯 新增 Entry/Step Span

问题背景：当 Agent 执行长程任务时，单个 Trace 中包含成百上千个 Span，导致调用链轨迹难以清晰观测。

解决方案

Entry Span

在 Agent 调用的入口处创建 Span，用于还原模型和用户的原始输入、输出，形成对话历史。确保处理的数据不受 System Prompt 或框架 Prompt 的干扰，获取最原始的客户请求。

Step Span

代表 Agent 在每次 ReAct 过程中的层次化表达。在"反思 → 工具调用 → 模型调用"的循环中，通过逐轮的 Span 结构清晰展示 Agent 的多轮行动、反思及执行结果。

排查流程

  1. 观察整体情况（如 Agent 执行包含 10 轮 ReAct 过程）
  2. 定位是哪一轮出现问题
  3. 深入分析该轮中具体是哪一步出错
  4. 通过逐轮 Span 结构快速定位问题

🛠️ 新增 Skill 语义

问题背景：现有 OTel GenAI 语义已覆盖 Agent、LLM、Tool 等 Span 类型，但缺少对 Skill 这一业务功能聚合层的抽象。

三大痛点

1

无法归因到功能域

性能抖动时只能看到 execute_tool 和 inference Span，无法快速判断是哪个功能域出了问题。

2

无法统计 Skill 健康指标

缺少 Skill 粒度的 P99 延迟、成功率、调用频率等度量。

3

多 Skill 并发时链路混淆

不同 Skill 的 LLM/Tool Span 在 Trace 树中无法区分归属。

语义建模

在 LoongSuite GenAI SemConv 中新增了一组 gen_ai.skill.* 属性：

gen_ai.skill.name        # Skill 名称
gen_ai.skill.version      # Skill 版本
gen_ai.skill.id           # Skill 唯一标识

当前阶段，这些属性附着在已有的 execute_tool Span 上，无需引入新 Span 类型即可快速落地。同时，基于集团业务落地了独立 invoke_skill Span 的方案。

🔬 新增 Token 级推理观测

核心价值：首次将推理引擎内部许多原本"黑盒"的过程拆解到 Token 粒度，打造可透视、可解释、可归因的白盒系统。

两大核心问题

⚡

性能异常

单个请求响应慢往往是因为某些 Token 生成慢，而 Token 生成慢大概率是其他请求并发干扰导致的。

🎯

精度异常

出现复读、答非所问、乱码等精度问题，往往从某个 Token 开始异常，后续 Token 受此影响持续出错。

Token 性能数据采集

在每个请求的每个 Token 粒度，采集进入迭代和退出迭代的时间戳：

# 核心指标
gen_ai.token.scheduling_time    # 调度时间
gen_ai.token.execution_time     # 实际执行时间
gen_ai.token.total_time          # 用户感知的总耗时
gen_ai.token.batch_size          # Batch 总请求数
gen_ai.token.batch_token_count   # Batch 总 Token 数

Token 精度数据采集

采集每个 Token 对应的候选 Top-K Token 的概率分布：

# 精度相关属性
gen_ai.token.top_k_candidates       # Top-K 候选 Token
gen_ai.token.top_k_probabilities    # 对应概率分布
gen_ai.token.selected_token         # 选中的 Token
gen_ai.token.sampling_parameters    # 采样参数（温度等）

典型案例

案例 1：慢 Token 定位，快速识别跨请求的资源干扰 ▼

线上某请求 TPOT 破线，Token 分析页面显示第 125 个 Token 花了 6.8 秒。通过引擎并发分析发现，请求 1 的 prefill 阶段（亮绿色块）花了 6s+，中断了请求 2 的 decode 过程（黄色块）。

根因：其他租户的请求 prefill 中断了当前请求的 decode 过程。
解决方案：PD 分离，避免不同请求的 prefill 和 decode 相互影响。
案例 2：Token 级观测，精准定位答非所问的根因 ▼

用户问的是医疗问题，但大模型回复了 LeetCode 解题答案。Token 分析页面显示首 Token 是 begin_of_sentence（BOS），这个特殊 Token 被用来分割毫无关联的两个语料。

结论：一旦出现了 BOS，接下来的回答就跟之前的 prompt 毫无关联，自然而然就答非所问了。后续发现产出 BOS 属于大模型的 badcase，解决方案是调整模型或等待后续模型版本优化更新。

🛠️ 使用 GenAI Utils 快速实现

LoongSuite GenAI SemConv 的工程化能力层

核心挑战：每个 GenAI 框架插桩库都需要实现一套完整的遥测采集逻辑，这些逻辑在不同框架插桩间高度重复。当语义规范迭代升级时，若每个插桩库各自维护，升级成本将成倍增长。
1

插桩层只做数据提取

各框架插桩库通过 Hook/Monkey-Patch 拦截框架调用，将数据填充到对应的 Invocation 数据对象中，不直接操作 OTel API。

2

GenAI Utils 统一收口

所有 Span 创建、属性挂载、Metrics 记录、Event 发送、Context 管理均由 ExtendedTelemetryHandler 内部完成。

3

规范升级只改一处

当 LoongSuite GenAI SemConv 新增字段或调整结构时，只需修改 GenAI Utils，所有下游插桩库自动生效。

API 使用示例（Python）

第一步：获取 Handler 单例

from opentelemetry.util.genai.extended_handler import get_extended_telemetry_handler

# 获取 ExtendedTelemetryHandler 单例
handler = get_extended_telemetry_handler()

第二步：选择 Invocation 数据类，填充业务数据

from opentelemetry.util.genai.invocation import AgentInvocation

# 创建 Agent 调用数据对象
invocation = AgentInvocation(
    agent_name="shopping_assistant",
    agent_id="agent_123",
    input_tokens=150,
    output_tokens=80,
    model="qwen-turbo"
)

第三步：使用 Context Manager 完成遥测输出

with handler.start_agent(invocation) as span:
    # 执行业务逻辑
    result = agent.execute(user_query)
    
    # 自动完成：
    # - Span 创建和结束
    # - gen_ai.agent.name 等属性挂载
    # - Duration 和 Token Usage Metrics 记录
    # - 异常捕获和错误状态设置

🎯 结语：从统一字段走向统一基础设施

GenAI 时代的可观测建设，已经从"为模型调用加日志埋点"演进到"为 Prompt、推理、检索、工具和 Agent 全链路建立统一语义"。

阿里巴巴与蚂蚁集团共建的核心意义：

  • 把标准化方向进一步工程化、平台化、规模化
  • 用统一语义降低业务接入成本
  • 用统一数据驱动观测平台、分析服务和治理能力的复用

最终目标：让所有使用该套规范的厂商和用户能够为快速增长的 GenAI 应用真正做到
