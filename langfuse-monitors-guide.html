功能概述

Langfuse Monitors 是一套面向生产环境的 LLM 应用监控解决方案。它允许你定义监控规则， 基于多种 Measure（度量指标） 追踪应用性能，并在指标异常时自动触发告警。

多维度度量

支持延迟、成本、质量、请求量等多种核心指标监控

实时告警

指标超过阈值时，通过邮件、Slack、Webhook 即时通知

灵活配置

支持按项目、环境、标签等多维度过滤和聚合

Monitor 工作流程

1
定义 Measure
选择要监控的指标类型
2
设置阈值
配置触发告警的边界条件
3
选择时间窗口
定义监控的聚合周期
4
配置告警
设定通知渠道和接收人

Measures 详解

Measure 是 Monitor 的核心，决定了你要监控什么。每个 Measure 都有特定的计算方式和适用场景。

Latency 延迟

核心指标 单位: 毫秒(ms)

监控 LLM 请求的端到端延迟，从发送请求到接收到完整响应的时间。这是评估用户体验和系统性能最直接的指标。

配置参数

aggregation 聚合方式
p50 / p75 / p90 / p95 / p99 / avg / max / min
threshold 阈值 (ms)
例如: 2000 (2秒)
operator 比较运算符
> / < / >= / <= / =
最佳实践

建议使用 p95 或 p99 来捕获尾部延迟，而不是平均值。平均值容易被少数极快请求拉低，掩盖真实的性能问题。

Cost 成本

核心指标 单位: USD ($)

追踪 LLM API 调用的累计成本。Langfuse 会自动根据 token 用量和模型定价计算每次请求的成本，Monitor 在此基础上进行聚合监控。

配置参数

aggregation 聚合方式
sum / avg / cumulative
timeBucket 时间分桶
1h / 6h / 1d / 7d
scope 范围
per-request / total / per-user
成本预警场景

设置每小时成本上限，防止因异常流量或模型切换导致费用激增。建议结合 per-user 范围追踪单个用户的消费情况。

Quality 质量评分

高级指标 单位: 0-1 或 0-100

基于 Langfuse Scores（评分系统）监控 LLM 输出的质量。可以追踪用户反馈评分、LLM-as-Judge 评分或自定义评估指标。

配置参数

scoreName 评分名称
user-feedback / llm-as-judge / custom
aggregation 聚合方式
avg / median / min / max
threshold 阈值
例如: 0.8 (低于此值触发告警)
评分来源

Quality Measure 依赖于 Langfuse Score 功能。确保你已经在应用代码中通过 langfuse.score() 或 UI 界面创建了评分数据。

Volume 请求量

核心指标 单位: 请求数

监控 LLM 请求的数量。可用于追踪流量波动、检测异常峰值（如 DDoS 或 bot 攻击）或确认服务可用性。

配置参数

aggregation 聚合方式
count / unique-count
groupBy 分组维度
trace / user / model / none
threshold 阈值
例如: 1000 (请求数/分钟)
流量监控技巧

同时设置 过高 和 过低 两个阈值。过高可能表示攻击或循环，过低可能表示上游服务中断或配置错误。

Error Rate 错误率

核心指标 单位: % (百分比)

追踪 LLM 请求的失败比例。包括 API 错误、超时、速率限制、内容过滤等所有导致请求未能成功完成的场景。

配置参数

errorTypes 错误类型
all / api_error / timeout / rate_limit / content_filter
aggregation 计算方式
percentage / count
threshold 阈值
例如: 5 (表示 5%)
错误分类监控

建议针对不同类型的错误创建不同的 Monitor。例如，单独监控 rate_limit，以便在接近 API 提供商限制时提前收到告警。

Throughput 吞吐量

高级指标 单位: tokens/second

监控 LLM 生成的吞吐量，即每秒生成的 token 数量。这个指标对于流式 (streaming) 应用特别重要，直接影响用户体验的流畅度。

配置参数

tokenType Token 类型
input / output / total
aggregation 聚合方式
avg / p95 / p99
threshold 阈值
例如: 50 (tokens/second)
流式场景优化

对于 ChatGPT 类应用，用户感知到的 "速度" 主要由 output tokens/second 决定。监控此指标并确保其高于用户可接受的最低阈值（通常 > 10 t/s）。

Configuration 详解

Monitor Configuration 是定义监控规则的核心。一个完整的配置包含基本设置、Measure 选择、阈值条件和过滤规则。

配置结构总览

{
  "monitor": {
    "id": "mon_prod_latency_p95",
    "name": "Production API Latency",
    "description": "Monitor P95 latency for production API calls",
    "status": "active",
    "projectId": "proj_xxxxx",

    "measure": {
      "type": "latency",
      "config": {
        "aggregation": "p95",
        "operator": ">",
        "threshold": 2000
      }
    },

    "window": {
      "type": "rolling",
      "duration": "5m",
      "step": "1m"
    },

    "filters": {
      "environment": ["production"],
      "tags": ["api", "critical"],
      "models": ["gpt-4", "gpt-4-turbo"]
    },

    "alert": {
      "channels": ["email", "slack"],
      "severity": "warning",
      "cooldown": "10m",
      "message": "P95 latency exceeded 2s threshold"
    }
  }
}

配置项详细说明

基础设置

字段 类型 必填 说明
id string 是 Monitor 唯一标识符，建议包含环境和指标信息
name string 是 Monitor 显示名称，建议清晰描述监控目的
description string 否 详细描述监控规则和用途
status string 是 Monitor 状态: active / paused / disabled
projectId string 是 所属项目 ID，Monitor 只会监控该项目的 Trace

时间窗口 (Window)

定义数据聚合的时间范围和评估频率。

Rolling Window (滑动窗口)

持续评估最近 N 分钟的数据，适合实时监控场景。

{ "type": "rolling", "duration": "5m" }
Fixed Window (固定窗口)

按固定时间间隔（如每小时）评估一次，适合周期性报表。

{ "type": "fixed", "duration": "1h" }

过滤条件 (Filters)

通过筛选器精准定位监控范围，避免噪音数据干扰。

environment

按环境过滤: production, staging, development

tags

按标签过滤，支持自定义业务标签

models

按模型过滤: gpt-4, claude-3 等

users

按特定用户 ID 过滤，适合重点客户监控

name

按 Trace/Span 名称过滤，支持通配符

metadata

按自定义元数据字段过滤

Alert 配置

当 Monitor 检测到指标超过阈值时，Alert 配置决定了如何通知相关人员。Langfuse 支持多种通知渠道和高级告警策略。

Email

发送邮件到指定地址，支持 HTML 格式。

{ "type": "email", "recipients": ["ops@company.com"] }

Slack

推送到 Slack 频道，包含图表和详细信息。

{ "type": "slack", "channel": "#alerts" }

Webhook

发送 HTTP POST 请求到自定义端点。

{ "type": "webhook", "url": "https://api.company.com/alerts" }

高级告警策略

Cooldown (冷却时间)

两次告警之间的最小间隔，避免告警轰炸。

"cooldown": "10m" // 10分钟内同一Monitor只告警一次

Severity (严重级别)

定义告警的紧急程度，影响通知方式和响应 SLA。

info warning critical

Escalation (告警升级)

配置多级告警，首次通知 on-call 工程师，长时间未处理则升级给主管。

{
  "escalation": {
    "levels": [
      { "after": "0m", "notify": ["on-call"] },
      { "after": "30m", "notify": ["team-lead"] },
      { "after": "1h", "notify": ["manager"] }
    ]
  }
}

实战示例

示例 1: 生产环境 P95 延迟监控

当生产环境的 P95 延迟超过 2 秒时告警

{
  "monitor": {
    "id": "prod_latency_p95",
    "name": "Production P95 Latency",
    "status": "active",
    "measure": {
      "type": "latency",
      "config": {
        "aggregation": "p95",
        "operator": ">",
        "threshold": 2000
      }
    },
    "window": {
      "type": "rolling",
      "duration": "5m"
    },
    "filters": {
      "environment": ["production"]
    },
    "alert": {
      "channels": [
        { "type": "slack", "channel": "#alerts" }
      ],
      "severity": "warning",
      "cooldown": "10m"
    }
  }
}

示例 2: 每小时成本上限监控

当每小时累计成本超过 50 美元时告警

{
  "monitor": {
    "id": "hourly_cost_limit",
    "name": "Hourly Cost Limit",
    "status": "active",
    "measure": {
      "type": "cost",
      "config": {
        "aggregation": "sum",
        "operator": ">",
        "threshold": 50,
        "currency": "USD"
      }
    },
    "window": {
      "type": "fixed",
      "duration": "1h"
    },
    "alert": {
      "channels": [
        { "type": "email", "recipients": ["finance@company.com"] }
      ],
      "severity": "critical",
      "cooldown": "1h"
    }
  }
}

示例 3: LLM 输出质量评分监控

当 LLM-as-Judge 评分低于 0.8 时告警

{
  "monitor": {
    "id": "quality_score_llm_judge",
    "name": "LLM Output Quality",
    "status": "active",
    "measure": {
      "type": "quality",
      "config": {
        "scoreName": "llm-as-judge",
        "aggregation": "avg",
        "operator": "<",
        "threshold": 0.8
      }
    },
    "window": {
      "type": "rolling",
      "duration": "30m"
    },
    "filters": {
      "tags": ["customer-facing"]
    },
    "alert": {
      "channels": [
        { "type": "slack", "channel": "#model-quality" },
        { "type": "webhook", "url": "https://api.company.com/quality-alert" }
      ],
      "severity": "warning",
      "cooldown": "15m"
    }
  }
}
